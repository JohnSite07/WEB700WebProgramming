Part 2: Updating Existing "get" Functionality to use Sequelize

 

Now that all of our sets exist on the database, we can refactor our existing code in the legoSets.js module to retrieve them. This can be done by following the below steps:

 

1.       You will notice that (after deleting the snippet to "bulkInsert" (from the end of Part 1), running node server.js will now cause an error something like: Cannot read properties of undefined (reading 'push').

This is because our initialize() method is still using the old "this.sets" and "this.themes". To fix this, delete the contents of initialize(), leaving an empty function.

Instead, invoke this.sequelize.sync() within a new returned Promise. If this.sequelize.sync() resolves successfully, then we can resolve the returned Promise, otherwise reject the returned Promise with the error.

If you try to run the server using node server.js now, we should see some SQL output to the console, and a confirmation that that the server has started correctly.

 

2.       Next, Change your code in the "getAllSets()" function to instead use the "this.Set" model (defined in the constructor()) to resolve the returned Promise with all returned sets (see: Operations (CRUD) Reference).

NOTE: Do not forget the option include: [this.Theme] to include Theme data when invoking "findAll".

 

 

3.       Change your code in the "getSetByNum(setNum)" function to instead use the "this.Set" model (defined in the constructor()) to resolve the returned Promise with a single set whose set_num value matches the "setNum" parameter (see: Operations (CRUD) Reference). As before, if no set was found, reject the Promise with an error, ie: "Unable to find requested set"

NOTE: Do not forget the option include: [this.Theme] to include Theme data when invoking "findAll". Also, remember to resolve with the first element of the returned array ([0]) to return a single object, as "findAll" always returns an array.

 

4.       Change your code in the "getSetsByTheme(theme)" function to instead use the "this.Set" model (defined in the constructor()) to resolve the returned Promise with all the returned sets whose "Theme.name" property contains the string in the "theme" parameter. This will involve a more complicated "where" clause, ie:

'$Theme.name$': {

[Sequelize.Op.iLike]: `%${theme}%`

}

}});

 

As before, if no sets were found, reject the Promise with an error, ie: "Unable to find requested sets"

NOTE: We have once again included the option include: [this.Theme] to include Theme Data.

 

5.       Look through your .ejs files for "set.theme" and instead replace it with: "set.Theme.name". This is because when "including" the Theme model in our above functions, we have added a full "Theme" object to the result objects (set / sets).

If you have been testing the functionality, you will have noticed that if we add a set with a "set_num" that already exists, we see an error returned to the console in plain text. Wouldn't it be better if we could show error messages in a way that is more consistent with the rest of the site?

 

To fix this, let's start by altering our "404.ejs" file to have a <%= message %> ejs variable instead of the hard-coded message (ie: "I'm sorry, we're unable to find what you're looking for.").

 

Now, in our server.js, whenever we write the code:

res.status(404).send(err); or res.render("404")

 

We can instead use:

 

res.render("404", {message: err}); or res.render("404", {message: "I'm sorry, we're unable to find what you're looking for."});

 

Finally, let's do the same thing for a new "500.ejs" file, ie:

1.       Create a new "500.ejs" file in the "views" folder

2.       Copy the contents of the "404.ejs" file and paste them into the new "500.ejs" file. Once this is complete, change the text "404" to "500"

3.       Now, if we wish to send an error (like the old 422 error when adding a set fails), we can instead use a more generic 500 error with the message:

res.status(500).render("500", {message: err});
