Part 3: Configuring EJS & Refactoring Existing Views

 

Since a major focus of this assignment is using EJS, the first step will involve correctly installing it and configuring it in our server.js file, ie:

 

·         Using npm to "install" the ejs package

·         Updating your server.js file to "set" the "view engine" setting to use the value: "ejs"

·         adding app.set('views', __dirname + '/views'); so that it works on Vercel

Once this is complete, we can now change all of our ".html" files (ie: "home.html", "about.html", "404.html", "addSet.html") to use the .ejs extension instead.  Unfortunately, this means that any code that we have in our server.js to "send" the html files, ie:

res.sendFile(path.join(__dirname, "/views/home.html"));


will no longer work (since the file should now be "home.ejs").  To remedy this, change all of your "res.sendFile()" functions to "res.render()".  For example, in the above case (for home.html) the new code would be:

res.render("home");

 

If you test your server now, you should not notice any changes in the browser (ie: the "about", "home", "addSet.html" and "404" pages are all rendering correctly).

 

Partial View – "navbar.ejs" 

 

Since we are using a template engine that supports "partial" views, we should take this opportunity to move some common HTML, used across all pages, into a "partial" view.  In our case, this is the navbar:

 

·         In the "views" directory, create a new folder called "partials"

·         Within the "partials" folder, create a new file: navbar.ejs

Since every one of our view files share the same navbar structure, copy the complete navbar code from one of your views (ie: "home.ejs") – in the demo, this is the <nav> ... </nav> element.  Paste this code (and any other code you might use for spacing, ie <br> etc.) in your "navbar.ejs" file.

 

Once this is complete, you can replace the navbar code in your views with the include statement for the "navbar" partial:

 

<%- include('partials/navbar') %>

 

You should be able to run your server now and see that it is once again running as before.  However, you will notice that the navbar element for "about" or "add to Collection", etc. is not highlighting correctly. 

 

To correct this, we should add a "page" parameter to our partial view code that matches the link in the navbar that we wish to highlight, for example, we can use:

 

<%- include('partials/navbar', {page: '/about'}) %>

 

for the "about" view.

 

NOTE: We must always include some value for "page", even if there's no corresponding link in the navbar (ie: we can use {page: ''} for 404, etc).

 

To make sure the correct element in the navbar is highlighted within the partial view, we need to update each of our <a>…</a> elements to conditionally add the 'active' class depending on the value of the "page" parameter.  For example, we can check if the "page" parameter matches the "href" value for the link, ie:

 

<li class="nav-item"><a class="nav-link" href="/about">About</a></li>

 

becomes

 

<li class="nav-item"><a class="nav-link <%= (page == "/about") ? 'active' : '' %>" href="/about">About</a></li>

 

 

Once this is complete, delete both the links for "Set" and "Theme" from the navbar

Part 6: Final Logic (deleting Sets & Rendering Themes in <select>)  

 

Our app should be functioning well and looking good by now.  The only things remaining are to implement the "deleting set" logic and address the <option>TODO</option> when creating new sets.  To begin, let's fix the "TODO":

·         First, in your /lego/addSet "GET" route, make sure you get all themes using the Promise-based "legoData.getAllThemes()" method and add them to the "addSet.ejs" view, ie: res.render("addSet", {themes: themes});

·         Next, in your addSet.ejs file, loop through all of the themes in the collection and render them as <option> elements in the select (instead of <option>TODO</option>), ie:

<% themes.forEach(theme=>{ %>

    <option value="<%= theme.id %>">

        <%= theme.name %>

    </option>

<% }) %>

·         Finally, we will need to update our new "set" to make sure we add the correct "theme" property back in the "POST" route /lego/addSet.  This can be done by first making sure we find the correct theme using:

let foundTheme = await legoData.getThemeById(req.body.theme_id);

Once we have the theme, we can update the req.body property with the correct name before adding it to the collection, ie: req.body.theme = foundTheme.name

 

We can now focus on the final piece – deleting a set.  This will involve:

 

·         Adding a single method to the LegoData class in the "legoSets.js" module called "deleteSetByNum(setNum)" where "setNum" is the setNum of the set to be deleted.  Here, we must:

o   Search through all the sets for the index of the set with the matching setNum from the parameter, ie:
let foundSetIndex = this.sets.findIndex(s => s.set_num == setNum);

o   If foundSetIndex isn't -1, then a set was found and it can be spliced out of the collection, using:
this.sets.splice(foundSetIndex,1); and the Promise can be resolved without any data

o   If foundSetIndex is -1, then a set was not found, and the Promise can be rejected with a meaningful error message

·         Once the method is ready, it can be used in a new GET route: /lego/deleteSet/:setNum

app.get("/lego/deleteSet/:set_num", async (req,res)=>{

       try{

              await legoData.deleteSetByNum(req.params.set_num);

              res.redirect("/lego/sets");

       }catch(err){

              res.status(404).send(err);

       }

});
